## 3. ENVIRONMENT VARIABLES

### Location: `C:\Projects\UDS_DASHBOARD\.env`

The .env file contains four API keys. Here is what each one is for and when to use it:

```env
# ============================================================
# PRIMARY — C1 / Thesys (dashboard generation, tool calling, UI rendering)
# Used by: backend/main.py for ALL C1 API calls
# This is the main engine. 90% of the tool runs on this key.
# Endpoint: https://api.thesys.dev/v1/embed
# Model: c1/anthropic/claude-sonnet-4/v-20251230
# ============================================================
THESYS_API_KEY=<your_c1_api_key>

# ============================================================
# DEEP ANALYSIS — Anthropic Claude (Opus for causal reasoning)
# Used by: Phase 2+ executive briefing generation
# NOT used in Phase 1 prototype
# Use case: When CEO asks "why" questions requiring multi-factor analysis
# Endpoint: https://api.anthropic.com/v1/messages
# Model: claude-opus-4-20250514 (or latest)
# ============================================================
ANTHROPIC_API_KEY=<your_anthropic_key>

# ============================================================
# ALTERNATIVE MODEL — OpenAI GPT-5.4
# Used by: Phase 2+ for comparison, artifact generation, alternative reasoning
# NOT used in Phase 1 prototype
# Use case: Board presentation generation, model quality comparison
# ============================================================
OPENAI_API_KEY=<your_openai_key>

# ============================================================
# FALLBACK — OpenRouter
# Used by: Only if primary Thesys API has outage or rate limit issues
# NOT used in Phase 1 unless Thesys is down
# Provides access to multiple models through single endpoint
# ============================================================
OPENROUTER_API_KEY=<your_openrouter_key>
```

### Frontend env: `frontend/.env.local`

```env
# The frontend proxies to the backend — it does not call C1 directly
# But if you need to call C1 from Next.js API routes:

```

## 8. FRAMEWORK DECISIONS — WHAT WE USE AND WHAT WE DON'T

### What we USE:
| Technology | Why |
|-----------|-----|
| **C1 / Thesys** | Generative UI — instant interactive dashboards from structured data |
| **FastAPI (Python)** | Governed backend — deterministic metrics, scoring, tool definitions |
| **Next.js** | Frontend host for C1Chat component |
| **SQLite** | Synthetic data store for prototype (swap to Azure SQL / PostgreSQL for production) |
| **C1Chat component** | Full-page conversational dashboard interface with thread persistence |

### What we DO NOT USE (and why):
| Technology | Why not |
|-----------|---------|
| **Microsoft Agent Framework 1.0** | Adds orchestration complexity with no Phase 1 value. C1's tool calling loop already handles single-agent governed tool access. Revisit only if Phase 3 requires autonomous multi-agent collaboration across domains. |
| **LangChain** | Adds abstraction layer over tool calling that C1 already handles natively. Direct OpenAI-compatible tool calling is simpler, more transparent, and easier to debug. |
| **AutoGen** | Multi-agent framework — not needed when we have one governed C1 interface with deterministic backend tools. |
| **Power BI Embedded** | This tool sits alongside Power BI, not inside it. The rest of the company uses Power BI. This is the CEO's personal C1 interface. |
| **Streamlit** | Limited UI capabilities compared to C1's generative UI. C1 produces richer, more interactive visualizations. |
| **Multi-model routing in Phase 1** | Adds complexity. Phase 1 uses THESYS_API_KEY for everything. Multi-model becomes relevant in Phase 2-3 for deep analysis (Opus) and artifact generation (GPT-5.4). |

## 11. C1 / THESYS TECHNICAL REFERENCE

The AI coder should also have access to the C1 documentation. Key pages:

- **How C1 Works**: Backend calls C1 API → receives C1 DSL → frontend renders with C1Component
- **Tool Calling**: Define tools as OpenAI-compatible functions → C1 calls them → uses results to generate UI
- **Streaming**: `stream: true` + `thesys_genui_sdk` for real-time UI rendering
- **Thinking States**: `write_think_item()` shows loading status during tool calls
- **C1Chat**: Full-page conversational interface with built-in thread management
- **Theming**: `themePresets.carbon` with dark mode for executive look
- **Forms**: Auto-generated from tool schemas (for future domain selector)
- **Multi-Step Flows**: `onAction` → new prompt → new UI (for drill-downs)
- **State Persistence**: `updateMessage` callback for saved views (Phase 2)
- **Artifacts**: Separate API endpoint for reports/slides (Phase 2-3)
- **MCP**: Standard protocol for external tool discovery (Phase 3)
